# xbrowser Shield 网络防护

## 目的

xbrowser 控制的是用户真实浏览器，会带上用户已登录网站的 cookie 和会话状态。Shield 网络防护默认启用，用于降低 prompt injection 诱导 agent 访问内网资源、云元数据端点或本机敏感页面的风险。

## 默认拦截范围

Shield 会拦截以下类型的访问：

- 私有/内网 IP 地址
- localhost / loopback 地址
- 云元数据端点（例如 `169.254.169.254`）
- 危险协议（例如 `file://`）
- DNS 解析到内网地址的目标（如策略实现支持）

## 用户可见回复规则

当用户询问“我能做什么 / 安全流程 / 注意事项”，或浏览器访问返回网络防护拦截错误时，最终回复必须使用自然语言说明流程，不得把 raw 命令作为用户操作步骤输出；包括状态、白名单、日志、引导、放行、撤销、运行、版本等所有 xb CLI 命令。

面向用户的安全流程：

1. 将拦截识别为安全策略，而不是浏览器故障或网络故障。
2. 说明被拦截的原因和访问该地址的风险。
3. 告知用户不会绕过网络防护，也不会代用户做安全决策。
4. 如果目标允许临时放行，要求用户明确确认操作和对象；确认后由 agent 内部执行对应操作。
5. 不主动建议关闭整个防护；只有用户明确要求关闭整体防护时，才说明高风险并发起确认。
6. 对云元数据端点、危险协议等不可放行目标，直接告知用户该 URL 不可访问。

禁止的最终回复形态：

- 不要说“你可以执行/请执行”后跟 raw 命令。
- 不要用具体 CLI 命令描述“查询状态/查看白名单/查看日志/排查原因”；改用“我会在内部查询最近的防护记录/拦截记录”。
- 加白、继续访问场景不要提到撤销白名单；撤销语义只用于用户明确要求撤销已有白名单或清理测试状态。
- guide / shield / status / list / logs / remove / enable / run / version 等 CLI 术语只用于内部实现，不是用户手动步骤。

## 拦截后的 agent 内部行为

CLI 细节只作为 internal/implementation reference。需要具体命令时查阅 [xb-cli-commands.md](./xb-cli-commands.md)，但不得在最终用户回复中展开：

1. 对可放行的 `host:port`，agent 内部发起加白确认流程。
2. guide 返回等待用户输入时，agent 展示自然语言风险说明和选项，停止等待用户选择。
3. 用户明确确认后，agent 只执行当前返回映射中明确给出的对应内部命令。
4. 用户要求关闭整体防护时，agent 内部发起关闭防护确认流程，但不得代用户确认。
5. 撤销已有白名单或清理测试状态时，agent 内部执行撤销操作；不要把撤销描述成加白或继续访问方式。

## 操作语义边界

Shield 相关命令分为“公开管理命令”和“确认引导命令”。agent 只能按每个命令的真实语义解释结果，不能因为用户想达成某个效果，就把一个操作说成相反含义。

| 操作 | 正确语义 | 禁止表述 |
|------|----------|----------|
| 临时放行 | 进入用户确认流程；用户明确确认后，才加入白名单 | 不得说成撤销白名单，不得说成已经放行 |
| 撤销放行 | 删除已有白名单条目 | 不得说成加白、放行或允许访问 |
| 关闭防护 | 进入关闭防护确认流程；用户明确确认后，才关闭 | 不得在模糊授权下关闭 |
| 启用防护 | 恢复/打开安全防护 | 不得说成关闭防护 |
| guide | 返回给最终用户的确认引导和选项 | guide 返回不代表操作已完成，不得说成已经加白或已经关闭 |

硬规则：撤销不是加白；启用不是关闭；guide 只是确认引导，不代表操作完成；不要把任何 shield 操作解释成相反语义。CLI/agent 执行参考不是最终用户回复模板，面向用户时必须用自然语言说明“需要确认/等待选择”，而不是把映射命令当成用户已经确认。

## 强确认边界

加白名单、关闭防护会改变网络安全边界，必须等待用户明确确认。有效确认必须同时具备三项：

1. **明确操作**：例如“临时放行”“加入白名单”“关闭防护”。
2. **明确对象**：加白时必须指明 `host:port`；关闭防护时必须指明整体网络防护。
3. **明确确认语义**：例如“我确认”“我了解风险并确认”。

以下表达都不构成确认，不能触发加白或关闭防护：

- “你自己看着办”
- “按你推荐”
- “随便”
- “别问我”
- “你决定”
- “先试试”
- “应该可以吧”
- “可以的话就做”

可接受的确认示例：

- “我确认临时放行 127.0.0.1:65531”
- “我确认把 192.168.1.10:8080 加入白名单”
- “我了解风险，确认关闭防护”

不可接受的确认示例：

- “按你推荐处理”——没有明确确认语义。
- “可以的话就放一下”——条件式授权，不是明确确认。
- “你决定要不要关闭”——把安全决策委托给 agent，不是用户确认。

即使 guide 返回了确认映射，也只有在用户的自然语言回复满足强确认边界后，agent 才能按当前映射执行对应内部命令。

## 面向用户的白名单流程

白名单用于允许特定 `host:port`，必须由用户明确确认。最终回复用自然语言说明：

1. 当前访问被安全策略拦截。
2. 该地址属于可申请临时放行的范围，并说明放行会允许浏览器访问该地址。
3. 要求用户明确确认“临时放行 + 具体 host:port”。
4. 用户确认后，由 agent 内部完成放行并继续访问。
5. 如用户之后要求撤销或清理测试状态，再用自然语言说明可撤销/移除该白名单。

最终回复不得把 raw 命令列为用户操作步骤。

## 面向用户的关闭防护流程

关闭整个防护风险很高，强烈不推荐。最终回复用自然语言说明：

1. 不主动建议关闭整体防护。
2. 如果用户只是想访问一个特定地址，优先说明“可对该地址单独发起临时放行确认”。
3. 只有用户明确要求关闭整体防护时，才说明风险并要求明确确认。
4. 用户确认后，由 agent 内部发起并执行关闭防护的确认流程。

最终回复不得把 raw 命令当作用户手动步骤展示。

## 用户确认边界

agent 不应根据已有命令名规律推断、构造或补全后续操作命令。需要用户确认的操作，唯一合法来源是当前 guide 返回的用户选择映射。

正确做法：

1. 执行内部 guide 工具。
2. 展示返回的决策点。
3. 等用户明确选择。
4. 只执行当前用户选择映射中明确给出的内部命令。

## 日志与排查

日志与状态查询是 agent 内部排查手段，不是最终用户回复模板。用户询问安全流程时，不要要求用户手动执行查询命令，也不要在“我会...”句式里展示具体 CLI 命令。

- status 用于 agent 内部确认防护是否启用、白名单数量和 24 小时拦截次数。
- list 用于 agent 内部确认某个地址是否已在白名单中。
- logs 用于 agent 内部查看最近拦截和防护相关事件。
- 面向用户描述排查动作时，写“我会在内部查询最近的防护记录/拦截记录”，不要写具体命令。
