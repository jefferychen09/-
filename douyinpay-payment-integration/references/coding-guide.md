# 抖音支付编码规范与注意事项

本文档为编码时参照的规范和注意事项，在需要为用户生成代码时遵循此规范。

***

## 一、代码结构规范

标准代码结构遵循以下顺序：

```
导入依赖 → 配置常量(占位符) → 核心逻辑 → 错误处理
```

### 1.1 占位符约定

所有敏感配置使用统一的占位符 并在代码中添加对应的注释，便于用户识别和替换：

| 占位符 | 注释与说明 |
| --- | --- |
| `YOUR_MCH_ID` | 商户号，注册并入驻抖音支付商家平台后获取 |
| `YOUR_APP_ID` | 应用 AppID（需先在商家平台-产品中心-AppID管理中绑定） |
| `YOUR_PRIVATE_KEY_PATH` | 商家公钥证书（RSA）对应的私钥文件路径 |
| `YOUR_MERCHANT_CERT_SERIAL_NO` | 商家公钥证书序列号（HTTP Authorization 头中的 `serial_no`） |
| `YOUR_PUBLIC_KEY_PATH` | 平台提供的抖音支付公钥证书文件路径（用于回调验签） |
| `YOUR_API_ENCRYPT_KEY` | 接口加密密钥（对称密钥，用于 AES-256-GCM 解密回调报文中的敏感信息） |
| `YOUR_NOTIFY_URL` | 支付结果回调通知地址（必须 https、公网可访问、不能携带参数，[详见文档](https://pay.douyinpay.com/wiki/66aa57118a7da602efb9bc2f/69bd121735060b0581d2ea46)） |
| `YOUR_RETURN_URL` | 前端跳转返回地址（H5支付场景） |

私钥和CSR可使用`bash scripts/gen_rsa_key.sh`帮用户生成并引用。

#### 客户端代码不使用占位符

客户端（iOS / Android / 鸿蒙）的调起支付代码**直接按官方文档示例的写法**，用字典字面量 + 示例值 + 中文注释展示各字段含义，不使用 `YOUR_XXX` 占位符。原因：

- 调起支付的参数由服务端预下单后完整返回，客户端只是透传
- 官方文档本身就是用示例值展示的，用户更容易理解
- 详见 `references/app-sdk-integration-guide.md` 第 14 节

### 1.2 注释规范

- 关键步骤使用中文注释
- 优先引用文档中的原始代码和参数说明，不自行编造
- 注释中引用对应的官方文档链接，格式：`// 参考: https://pay.douyinpay.com/wiki/xxx/yyy`
- 函数/方法级注释说明入参、出参和可能的异常

### 1.3 安全规范

- **私钥**：从文件或环境变量读取，严禁硬编码到代码中
- **回调验签**：收到支付回调后必须先验签，再处理业务逻辑
- **不信任前端**：不依赖前端传来的支付结果，以服务端回调为准
- **幂等处理**：回调接口需要支持幂等，避免重复处理，不仅回调需要幂等，下单也要防重复创建
- **参数过期处理**：注意各参数的有效时间和过期时间，关注是否需要重新获取
- **要素一致性校验**：回调/查单验签通过后，须校验关键字段与本地订单一致（商户号/应用标识/订单号/金额/币种等），不一致需告警

### 1.4 README.md 规范

编码结束前**必须**为用户完善 README.md 文件，具体要求

1. 检查项目中是否已有 `README.md` 文件，如果已有则新建一个 `DOUYINPAY_README.md` 文件来说明，反之则新建 `README.md` 文件并说明
2. 说明需包含：
  - 生码的代码文件结构说明
  - 引入的关键依赖
  - 关键配置说明
  - 项目初始化和前后端启动命令

### 1.5 工程化结构要求

- 采用模块化、原子化设计，不要把所有逻辑堆砌到一个文件里

## 二、SDK 支持

### 2.1 服务端 SDK

目前提供Go、Java、PHP三种语言的官方SDK。在生成项目依赖配置（如 `go.mod`、`pom.xml`、`composer.json`）时，**必须获取真实的最新版本号（如 Maven 坐标版本、Go 仓库 Tag 等）**。

**获取真实版本号的指令动作：**

1. 优先调用 `search_docs` 工具（即 `bash scripts/search_docs.sh "<query>"`）查阅官方接入文档中的依赖引入示例，提取其中的版本号。
2. 若文档中未明确标出最新版本，**必须**前往对应的代码仓库获取最新 Release 版本。例如：
   - Go：通过搜索 `github.com/douyinpay/douyinpay-go` 查找最新的 Tag。
   - Java：通过搜索 Maven 中央仓库 `central.sonatype.com/artifact/io.github.douyinpay/douyinpay-java` 获取对应的最新坐标版本。

注意：**严禁自行编造或幻觉产生版本号**（**绝对不要使用** `v0.0.0-latest`、`1.0.0` 等未经查证的错误占位符）。必须在输出代码前完成版本号的准确查询。

### 2.2 客户端 SDK

目前提供 iOS、Android、鸿蒙（HarmonyOS）三种平台的官方客户端 SDK，具体 SDK 下载地址及集成方案可以调用文档检索工具查询。

#### 2.2.1 原生 APP 支付优先级

- 当用户目标为已有 iOS / Android / 鸿蒙原生工程中的 APP 支付接入时，优先走原生客户端 SDK 接入流程
- 命中 APP SDK 接入场景时，先读取 `references/app-sdk-integration-guide.md`
- `references/app-sdk-integration-guide.md` 是 APP SDK 接入的唯一正文来源，本文件不再重复维护平台细则

#### 2.2.2 客户端最小红线

- 客户端不生成签名逻辑，只消费服务端返回的完整调起参数
- 客户端结果仅供参考，最终支付状态以服务端通知/查询为准
- 所有已有支付实现参考统一表述为"三方支付"，不主动输出任何具体品牌名
- 生成的接入代码必须与官方文档示例代码保持同等复杂度——文档中没有的逻辑，代码中也不加

### 2.3 开发语言与规范确认

- 用户未明确时，优先使用已有项目中的语言
- 遵循用户侧研发规范要求（如有）

### 2.4 无官方 SDK 的语言

对于官方 SDK 的语言：

- 使用 HTTP 协议直接调用抖音支付 API
- 需要自行实现签名与验签逻辑
  - 如何构造请求签名：https://pay.douyinpay.com/wiki/66aa57118a7da602efb9bc2f/66aa574888f38d02f9de20da
  - 如何验证签名：https://pay.douyinpay.com/wiki/66aa57118a7da602efb9bc2f/66aa57df29710302ee143340
- 可以使用 `bash scripts/gen_rsa_key.sh` 生成密钥对

## 三、典型编码流程

### 3.1 服务端接入流程（以 Go 为例）

```
1. 安装 SDK → go get github.com/douyinpay/douyinpay-go
2. 初始化客户端（配置 AppID、商户号、私钥路径）
3. 构造下单请求（商品信息、金额、回调地址）
4. 调用下单接口
5. 处理下单响应（获取支付参数/二维码链接）
6. 实现回调接口（验签 → 业务处理 → 返回成功）
7. 实现查询订单接口（可选，用于主动轮询）
8. 实现退款接口（可选）
```

### 3.2 客户端接入流程（以 APP 支付为例）

命中客户端 APP 支付接入场景时，**必须**先读取 `cat references/app-sdk-integration-guide.md`，按其中的完整流程执行。

## 四、常见注意事项

### 4.1 签名相关

- 签名算法：RSA-SHA256
- 私钥格式：PKCS#8 PEM（`bash scripts/gen_rsa_key.sh` 生成的即为此格式）
- 签名字符串拼接顺序严格遵循文档说明，字段顺序不可变更
- 验签使用抖音支付公钥证书，需提示用户商家平台下载

### 4.2 回调处理

- 回调地址必须是公网可访问的 HTTPS 地址
- **验签与解密**：收到回调后务必先对 HTTP 请求头中的签名信息进行验签；通知数据（`resource`）使用了 AES-256-GCM 加密，需使用接口加密密钥解密后才能获取业务明文。回调验签通过后，必须做“要素一致性校验”：订单号、金额、商户号/应用标识等必须与本地订单一致
- **响应要求**：处理成功时，**必须返回 HTTP 状态码 200 或 204，无需返回任何应答报文（body 为空即可）**；处理异常时，应返回 4xx 或 5xx 状态码并返回应答报文。
- **幂等与并发控制**：同一笔订单的回调可能重复、乱序到达，商户系统必须支持幂等处理。建议在检查/更新业务数据状态前，采用**数据锁进行并发控制**，避免函数重入造成数据混乱，避免重复入账/重复发货。
- **重试机制**：如果响应超时或 HTTP 状态码非 200/204，抖音支付会按策略重试（间隔 15s 至 6h 不等，总计约 24 小时 4 分钟）。
- **须有“回调缺失兜底”**：若长期未收到回调，应有定时任务或人工手段触发查单，避免订单卡死

### 4.3 金额处理

- 接口金额单位为 **分**（整数），如 1 元 = 100
- 确保金额类型为整数，不要传浮点数
- 前端展示时需转换为元，且注意**浮点数精度问题**
- 对账单位换算：若账单金额单位与接口金额单位不同，必须统一单位后再对账

### 4.4 常见错误码

| 错误码 | 常见原因        | 排查建议                   |
| --- | ----------- | ---------------------- |
| 401 | 签名错误 / 证书无效 | 检查私钥是否正确、签名字符串拼接是否符合文档 |
| 400 | 参数缺失或格式错误   | 核对必填参数和类型              |
| 403 | 权限不足        | 检查商户是否开通了对应产品          |
| 500 | 服务端内部错误     | 稍后重试，持续报错联系技术支持        |

其它错误可使用`search_docs`工具（`bash scripts/search_docs.sh "错误信息"`）检索相关信息。

### 4.5 代码规范

#### 前端代码规范

- 注意分离js/ts/tsx、css、html，尽量避免内联 style 和 onclick
- 前端优先使用Typescript
- 如无必要，不引入任何构建工具

#### 后端代码规范

- Go推荐html/template + htmx实现
- PHP推荐Blade/Twig + htmx实现
- Java推荐Thymeleaf + htmx实现

## 五、代码生成行为规范（必须遵守）

### 5.1 输出形态与完整性要求

- **Demo 模式（从零跑通）**：当用户提供的是空目录/明确要一个可运行的示例工程时，输出必须包含所有可运行代码文件（不允许只给 Markdown 指南代替代码）。若同时包含客户端与服务端代码，二者必须放在同一个项目目录下。
- **存量工程模式（最小侵入式改造）**：当用户提供现有项目工程时，默认在用户现有工程结构内完成最小改造，不额外生成独立 demo 或大规模重构；若因信息不足无法安全落盘，先追问确认目标模块/文件再继续。
- **证书与密钥**：若用户尚未生成商户私钥/CSR，可调用 `bash scripts/gen_rsa_key.sh --out-dir ./certs` 生成（仅 RSA），并提示 CSR 上传以申请商家公钥证书（RSA）；若用户已有证书密钥材料，不覆盖用户现有文件。

## 六、联调与测试

为了确保接入抖音支付安全、顺利上线，需注意：

- 抖音支付目前未提供测试或沙箱环境，请务必注意资金安全（扣款、结算等均为线上环境数据）
- 若为demo或测试场景，订单金额应为最小的1分钱
- H5支付 与 JSAPI支付 需先在 商家平台-产品中心-开发配置 中配置H5支付域名与JSAPI授权目录
