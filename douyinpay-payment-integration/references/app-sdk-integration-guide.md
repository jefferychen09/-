# APP SDK 接入主参考（客户端）

> 本文件是 APP SDK 接入的流程编排与检索校验参考，不是官方事实源。`SKILL.md` 仅保留入口/阻塞检查点/澄清顺序与能力边界；`coding-guide.md` 仅保留共性编码约束。
> 涉及 SDK 版本号、API 名称、方法签名、配置项、下载地址时，必须以本次运行时检索并获取的官方 Markdown 原文为唯一依据。

## 0. 执行顺序速查（锚点）

`读取官方文档` → `§2bis 三方支付只读识别` → `按第 3 节完成澄清` → `回显本次命中维度` → `按第 5–11 节执行（落位参考 §11.2 决策表）` → `按第 12.1 节四类归档输出`

## 1. 适用范围

当用户要在 iOS / Android / 鸿蒙原生工程中接入抖音支付客户端 SDK，并涉及以下任一事项时使用本文件：

- 集成 SDK（包管理 / 离线包）
- 调起支付
- 处理支付结果回跳
- iOS `Scheme` / `Universal Links` 接入分支
- 已接入三方支付时参考其既有代码组织方式落位

合规要求：

- Skill 主动对外输出时，所有已有支付实现统一表述为 `三方支付`，不主动输出任何具体品牌名
- 用户自行提供的代码、日志、文件路径中出现具体品牌名时，允许在引用用户事实的语境内按原文保留，但不得由此派生新的品牌相关描述
- 第 11.1 节识别锚点仅用于 Skill 侧内部判断，识别结果回显给用户时隐去品牌关键词，只回显"依赖坐标、文件路径、命中锚点类别"

## 2bis. 三方支付只读识别（澄清前置）

第 2 节文档读取完成后、进入第 3 节澄清前，执行一次低成本只读识别，用于：

- 为澄清项 3（是否已接入三方支付）提供证据
- 为澄清项 5（目标文件）提供候选排序依据

允许读取的文件范围严格等同于 §12.2 能力边界，不得扩张：

- iOS：`Podfile`、`Podfile.lock`、`Info.plist`、`*.xcodeproj/project.pbxproj`
- Android：`settings.gradle(.kts)`、根与 module 级 `build.gradle(.kts)`、`AndroidManifest.xml`
- 鸿蒙：`oh-package.json5`、`module.json5`

识别动作：

1. 从依赖清单中提取疑似支付 SDK 依赖（按 §11.1 识别锚点）
2. 从源码索引/配置中提取疑似支付职责点文件（按 §11.1 识别锚点）
3. 将以上结果整理为结构化回显：
   - 疑似三方支付通道清单（依赖坐标 + 来源文件）
   - 疑似支付职责点候选（文件路径 + 命中锚点类别）
   - 明确标注"以上为只读识别结果，需要你确认后再继续"

识别规则：

- 只读识别，不做任何改动、下载、落盘
- 识别到 0 项时直接标注"未识别到三方支付痕迹"，不再追问通道
- 识别到多项时按 §11.2 落位决策表的候选排序规则呈现
- 识别失败或命中能力边界外的文件需求时，按第 4 节阻塞检查点处理
- 本节输出不替代第 3 节的用户确认，仅作为确认问题的上下文

## 3. 接入前澄清（按顺序）

本阶段只收集用户确认，不做任何下载、配置、落盘、生码、方案落位等实际动作。
任一澄清项未被用户显式确认时，按第 4 节阻塞检查点处理。

澄清项：

1. **平台确认**：iOS / Android / 鸿蒙
2. **集成方式确认**：包管理工具 / 离线包
3. **是否已接入三方支付确认**
   - 先回显 §2bis 的只读识别结果
   - 若识别到疑似三方支付通道：追问"以上通道是否属于你当前工程的有效三方支付？是否有未被识别到的通道？"
   - 若未识别到：追问"是否确实未接入任何三方支付？"
   - 用户回答前不得进入执行阶段
4. **目标模块 / target 确认**：出现多个候选时必须先确认归属
5. **目标文件确认**：用于参考三方支付落位
   - iOS：见第 11 节
   - Android / 鸿蒙：不允许默认选择目标文件
   - 候选列表必须基于 §2bis 只读识别结果，并按 §11.2 决策表的候选排序规则排序
   - 只读识别到的候选项在用户显式确认前，按第 4 节阻塞检查点处理
6. **iOS 回跳方式确认**（仅当平台为 iOS 时确认，Android / 鸿蒙跳过）
   - 若用户已明确按 `Universal Links` 接入：追问完整 Universal Link 链接
   - 若用户未说明：追问"Scheme 还是 Universal Links"

## 4. 阻塞检查点（全局唯一来源）

本节是全流程的停止规则事实源。其他章节提到"停止/阻塞/等待确认"时，统一回到本节判断。
阻塞分两级：**硬阻塞**立即暂停并等待用户确认；**软阻塞**转入最小侵入式方案并在输出中显式声明。

### 4.1 硬阻塞（命中任一条即暂停，不得自行推断后继续）

1. 任一必填澄清项缺失、用户尚未确认，或用户尚未明确同意进入执行阶段
2. 只读识别得到的候选项尚未被用户显式确认
3. 需要**结构性改动**时（定义见 §11.3）：
   - 修改三方支付分发器/注册表的既有签名或协议
   - 替换或迁移既有支付入口
   - 删除现有支付通道的关键代码或配置
4. 用户提供信息与代码现状冲突且无法调和
5. 关键文档读取失败、内容冲突，或读完后仍无法支持唯一判断
6. 平台最终产物与当前平台不匹配（第 7 节下载校验失败）

### 4.2 软阻塞（转最小侵入式方案，不停止）

1. 出现多个模块、target、ability、入口文件或落位文件候选：
   - 按 §11.2 决策表排序后呈现给用户，默认不选择，但允许继续生成**不依赖具体落位**的产物（如独立的 util/工具类代码）
2. 工程结构与文档示例不完全一致，但属于**追加式改动**（定义见 §11.3）：
   - 沿用工程既有风格，输出 diff / patch 片段而非全量替换
3. 调整既有接入方式但不触及三方支付架构的核心协议：
   - 允许继续，但必须在输出中标注"本次改动影响的文件清单"并由用户最终确认

软阻塞下的输出必须显式声明"以下为最小侵入式方案，基于 <依据>，请确认后再合入"。

下载规则：

1. 从官方客户端 SDK 集成文档的"4、离线包下载"章节获取对应平台的下载地址
2. 必须从文档原文的 `<a>` 标签提取下载信息：`href` 作为真实下载 URL；若存在 `filename="..."`，优先作为保存文件名
3. 下载链接显示为 `...image.image` 或其它非常规后缀时，不能仅根据 URL 后缀判断失败
4. 同一平台有多个候选附件且无法唯一判断时，按第 4 节阻塞
5. 直接下载到用户已确认的工程内落地目录，不先落到系统下载目录；`Frameworks/`、`目标 module/libs/` 等只能作为候选目录
6. 下载完成后回显来源文档地址、真实下载 URL、保存文件名和最终工程内绝对路径
7. 如需解压，只能在目标工程目录内进行；完成后清理压缩包和中间目录，只保留最终引用产物

平台产物处理与校验：

1. iOS：zip 解压后只保留 `xcframework`
2. Android：最终保留 `aar`
3. 鸿蒙：最终保留 `har`
4. 必须结合响应头或文件内容判断是否为真实二进制产物，不能只看 URL 后缀
5. 平台最终产物与当前平台不匹配、或最终未得到对应产物时，视为失败；失败后不继续生成引用代码或工程配置，改为输出失败原因和下一步建议

## 8. 平台维度：iOS（基础配置 / Scheme 分支 / Universal Links 分支）

### 8.1 iOS 基础配置（两种分支都必须）

先基于已读取的「客户端 SDK 集成说明」整理一份 **iOS 必配清单**，至少覆盖：

1. `Info.plist` 配置，如 `LSApplicationQueriesSchemes`、`CFBundleURLTypes`
2. 系统库和 framework 检查项
3. `Build Settings` / `entitlements` 等文档明确提到的配置

执行时只按本次运行时读取到的官方文档原文处理，不在 skill 中预置固定名单。

处理流程：

1. 先从文档提取完整清单
2. 再按"是否能稳定落盘"分两类处理：
   - `Podfile`、`Info.plist`、`entitlements` 等仓库内文件，在 target 归属明确时直接修改，归类为 `我已自动完成`
   - Xcode `Link Binary With Libraries`、`Frameworks, Libraries, and Embedded Content` 里的系统库和 framework，不自动改 `project.pbxproj`，统一归类为 `请你手动完成`
3. 存在多个 target 且无法安全判断归属时，按第 4 节阻塞

按集成方式补充输出：

- **CocoaPods 集成**：先完成文档要求的 `Podfile` 源配置和依赖声明；文档要求的系统库或 framework 整理成手动检查清单给用户

### 8.2 iOS Scheme 分支

### 8.3 iOS Universal Links 分支

与 Scheme 分支的差异点：

1. 必须先收集完整 Universal Link 链接，解析出域名（用于 applinks 配置）
2. UL 模式注册 API 的完整签名（「客户端 SDK 集成说明」的"使用 Universal Links 回跳商户 APP"章节）
3. UL 结果回调 API 的完整签名及所在生命周期方法（含 AppDelegate 和 SceneDelegate 两种写法）
4. 回退要求：即使走 UL，也必须保留 callbackScheme 与回传 scheme 配置，用于低版本降级

生命周期方法共存规则：

- 若 §2bis 识别到 `application:continueUserActivity:restorationHandler:` 或 `scene:continueUserActivity:` 已被既有三方支付占用：
   - 输出 diff 风格 patch，仅新增按 UL `host / path` 判定走抖音支付 UL 回调的分支
   - 保留既有三方支付 UL 分支不动
- 若方法不存在：按前述 1–4 条正常生成全量示例（含 AppDelegate 和 SceneDelegate 两种写法）

输出需覆盖：注册、拉起支付、回跳处理 3 个动作点。

## 11. 工程现状维度：三方支付参考策略

若用户已接入三方支付，优先参考其现有支付代码组织方式，而不是强行套标准模板或预设其架构形态。

说明：

1. 三方支付工程可能存在多层封装与业务魔改（统一入口、分发中心、注册表、工厂、适配层等），本文件不对其结构做假设
2. Skill 的目标是"在现有工程结构内完成接入"，而不是为了接入抖音支付去重构用户现有三方支付架构

收集与落位原则：

1. iOS：以"职责点"而非"固定文件类型"来收集与落位（职责点可能重合到同一个文件）：
   - 调起支付职责点
   - 注册职责点
   - 结果处理职责点

   生命周期入口文件只作为线索，不假设一定是唯一落点。

2. Android / 鸿蒙：只做只读识别并整理候选文件（通常是支付入口或分发中心相关文件），不得默认选择其中任一文件作为最终落位点。

停止条件统一按第 4 节阻塞检查点处理。若改动会触及大范围重构或影响既有三方支付架构稳定性，转为提供最小侵入式方案建议，并由用户确认后再继续。

### 11.1 识别锚点清单（§2bis 与 §11 共用）

以下锚点仅用于 Skill 侧内部识别，**不在对用户输出中提及任何具体三方支付品牌名**（§1 合规要求）。

#### iOS 识别锚点

依赖清单（`Podfile` / `Podfile.lock`）：

- 命中支付相关 pod 名关键词：`Pay`、`Payment`、`Cashier`、`Checkout` 等通用英文词根
- 命中以支付品牌为前缀的 pod（按 pod 名字符串整体识别，不在输出中回显品牌名）

源码（只读 `*.xcodeproj/project.pbxproj` 的文件索引，不进入源文件内容）：

- 文件名包含 `Pay`、`Payment`、`Cashier`、`Checkout`、`PayManager`、`PayService`、`PayChannel`、`PayRouter` 等词根
- `AppDelegate.swift/m/mm`、`SceneDelegate.swift` 作为生命周期入口候选

#### Android 识别锚点

依赖清单（根与 module `build.gradle(.kts)`）：

- `implementation` / `api` 行中包含支付相关 artifact 关键词

源码索引：

- 文件名包含 `Pay`、`Payment`、`Cashier`、`Checkout`、`PayManager`、`PayService`、`PayChannel`、`PayDispatcher` 等词根

配置（`AndroidManifest.xml`）：

- `<activity>` 带 `*.pay.*` 等 callback activity 声明
- intent-filter 中的 scheme 命中支付相关前缀

#### 鸿蒙识别锚点

依赖清单（`oh-package.json5`）：

- `dependencies` 中包含支付相关 HAR 名关键词

配置（`module.json5`）：

- `abilities` 中命名包含 `Pay` / `Payment` / `Cashier` 词根的 ability
- `skills` 中 uri scheme 命中支付相关前缀

#### 三个平台通用职责点归类

- **调起支付职责点**：调用 SDK `pay` / `startPay` / `sendReq` 等 API 的代码所在文件
- **注册职责点**：调用 SDK `register` / `init` / `setXxxAppId` 等 API 的代码所在文件
- **结果处理职责点**：实现 SDK 回调协议/接口/监听的代码所在文件

三类职责点可能重合到同一个文件（见 §11.2 决策表）。

### 11.2 落位决策表

根据 §11.1 识别结果，按下表决定本次抖音支付代码的落位行为：

| 职责点分布 | 现有入口状态 | 落位行为 | 阻塞等级 |
|---|---|---|---|
| 三职责重合于同一"统一入口/分发器"文件 | 唯一 | 在同一文件内追加抖音支付分支，不拆新文件 | 直接执行 |
| 三职责分散于多个文件，各自职责清晰 | 各唯一 | 按职责点分别追加到对应文件 | 直接执行 |
| 职责点存在但有多个候选文件 | 多候选 | 按排序规则呈现给用户确认 | 软阻塞（§4.2） |
| 识别到三方支付依赖但未定位到职责点 | 未知 | 追问用户指明目标文件，或建议新建与其同级的文件 | 软阻塞（§4.2） |
| 未识别到三方支付 | 无 | 按平台文档示例标准落位，不强造抽象层 | 直接执行 |
| 需要修改分发器协议/签名才能接入 | 任意 | 改为输出"建议改造方案 + 影响面" | 硬阻塞（§4.1） |

候选文件排序规则（用于"多候选"场景）：

1. 三职责重合度高的文件优先
2. 文件名命中锚点数量多的优先
3. 与生命周期入口（AppDelegate 等）有直接引用关系的优先
4. 同排序下按路径字母序稳定排序

### 11.3 追加式 vs 结构性改动判定

用于 §4 阻塞分级和 §11.2 决策表的统一判定标准。

**追加式改动**（软阻塞或直接执行）：

- 仅新增文件，不修改既有文件的导出符号
- 修改既有文件时只做追加，不删除/重命名既有方法、协议、常量
- 在既有分支语句中新增 case / 分支，既有分支保留原行为
- 在既有依赖清单中追加一行依赖，不升降级既有依赖版本

**结构性改动**（硬阻塞）：

- 修改三方支付分发器/注册表的既有签名、协议、注册顺序
- 删除或重命名既有支付入口、回调方法
- 升降级既有三方支付 SDK 版本
- 将既有单通道分发改造为多通道路由（需要用户主动授权）

最小侵入式方案 = 用"追加式改动"能达成抖音支付接入目标的最小改动集合。

## 12. 全局规则（横切）

本节不参与路径选择，但对所有执行路径持续生效。优先级顺序：

1. 第 4 节阻塞检查点优先级最高
2. 用户已明确确认的澄清结果高于默认示例和候选项
3. 第 5–11 节执行维度决定本次具体执行路径
4. 本节约束输出方式与生码方式

### 12.1 输出模板（四类归档）

输出统一分为四类：

1. `我已自动完成`：工程内、仓库内、可稳定落盘的配置
2. `我已为你生成`：可产出结果物但需要用户托管/上传/平台配置
3. `我需要你补充后继续自动完成`：理论可自动完成，但本次 target/module 归属不明、候选过多、结构异常导致无法安全落盘；也包括 §4.2 软阻塞下已生成最小侵入式方案但需要用户确认落位的产物
4. `请你手动完成`：仓库外平台后台、域名/AASA 托管、或 agent 无法代做

同一任务的不同结果物可以分别归入不同类别。若某项结果在执行过程中从"可自动完成"变为"归属不明或风险升高"，应重新归类为 `我需要你补充后继续自动完成`。

### 12.2 能力边界

1. 只允许一次低成本只读定位（iOS `Podfile`、`Info.plist`、`.xcodeproj`；Android `settings.gradle`、`build.gradle`、`build.gradle.kts`；鸿蒙 `oh-package.json5`、`module.json5`）；超出范围的识别立即追问
2. iOS/UL 需要用户提供完整 Universal Link；AASA 托管属于仓库外事项，归类为 `我已为你生成` / `请你手动完成`

### 12.3 生码原则

1. 客户端不生成签名逻辑，只消费服务端返回的完整调起参数
2. 调起支付参数的示例代码**直接按官方文档「APP 调起支付」页面的示例写法**，用字典/Map 字面量列出各字段、填入示例值和中文注释；同时在注释中说明"以上参数均由服务端预下单接口返回，客户端不做签名计算"
3. 客户端结果仅供参考，最终支付状态以服务端通知/查询为准
4. 优先修改用户现有原生工程文件，不生成独立的客户端 demo 或桥接工程
5. 生成的接入代码必须与官方文档示例代码保持同等复杂度——文档示例是直接调用，就直接调用；文档中没有的逻辑，代码中也不加
6. 若用户项目已有三方支付模块，沿用其现有风格落位；没有参考时，不主动新造抽象层
