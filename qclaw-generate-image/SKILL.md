# AI 生图 — 文字生成图片 & 图片生成图片

根据用户的文字描述生成高质量图片，或基于参考图片生成新图片。脚本内部自动处理任务提交、等待和图片下载，鉴权由后台网关自动处理。

### ⚠️ 轮询与任务完成规则（CRITICAL — 必须严格遵守）

生图脚本执行后会自动轮询后端任务状态，**通常需要 20-60 秒**才能完成。你**必须**遵守以下规则：

1. **NEVER 在图片生成任务完成前结束当前 turn**。脚本运行期间你必须持续等待，直到脚本返回最终 JSON 结果（`success: true` 或 `success: false`）。
2. **NEVER 主动放弃等待**。即使脚本运行时间较长（可能超过 30 秒），你也不能中途输出"生成中，稍等片刻…"之类的文本然后结束 turn。你必须在**同一个 turn** 内完成"执行脚本 → 等待结果 → 输出模板"的完整流程。
3. **NEVER 依赖自动唤醒机制**。不要期望结束 turn 后系统会自动回调通知你结果——本技能要求在单次 turn 内同步完成所有步骤。
4. 如果 process 工具返回了"进程仍在运行、暂无输出"之类的中间状态，你**必须继续调用 process 工具轮询**，直到获得最终输出。绝对不能因为暂时没有输出就判定任务结束。

**正确流程**：执行脚本 → 持续轮询等待（可能多次调用 process 工具）→ 收到最终 JSON → 按模板输出结果。

## 触发决策（CRITICAL — 先判断「要不要画」，再谈「怎么画」）

本技能只在用户**意图是「生成一张新图像」**时触发。触发判断不是简单匹配关键词，而是理解意图。按以下三步决策：

### 第一步：正向信号 → 触发

命中任一类即视为生图意图（不限于字面触发词）：

| 信号类型 | 示例 |
|---------|------|
| 显式祈使 | 画/画一张/帮我画/生成图片/生成一张/AI作画/draw/generate image |
| 隐式口语 | 「来张…的图」「整一个 logo」「弄张封面」「做一张配图」「出一张宣传图」「我想要一张…壁纸」 |
| 产物是图像 | 海报、壁纸、头像、表情包、配图、封面、插画、吉祥物、Logo 等「最终交付物本身是图片」的需求 |
| 多轮延续 | 上一轮刚生成过图，用户说「再来一张」「换个风格」「换成黑猫」「把刚才说的那个画出来」 |
| 提供参考图 + 改动 | 「把这张变成油画」「参考这张图改成赛博朋克」「基于这张图加雪景」「以图生图」（→ 图生图模式） |

> 原则：**枚举词只是提示，不是上限。** 只要用户最终想要的是「一张图片」，即使没说「画」字也应触发。

### 第二步：负向排除 → 即使命中上面的词，也**不要**触发

以下场景**禁止**调用生图脚本，应交给对应能力或正常回答：

| 场景 | 典型用户输入 | 正确去向 |
|------|------------|---------|
| **图片理解/分析**（最高频误触） | 用户**发了图**问「这张图什么风格 / 里面有什么 / 这是什么品种 / 帮我看看 / 描述一下 / 提取文字」 | **image 工具**，不是图生图 |
| **「画」的非绘画义** | 画**表格**、画**流程图**/架构图/思维导图、画**重点**（划重点）、「这幅画的作者是谁」、成语（画蛇添足） | 表格/图表/文本/问答 |
| **「生成」非图像产物** | 生成周报/文档、生成代码、生成 PPT、生成测试数据、做成图表（数据可视化） | 对应的文档/代码/数据能力 |
| **谈论而非需求** | 「你会画图吗」「AI 生图用什么模型」「怎么写好提示词」「MJ 和 SD 哪个好」 | 正常问答，不生图 |
| **纯文本任务** | 翻译、写邮件、闲聊、查询，无任何图像信号 | 正常回答 |

> **关键判别规则（务必记牢）**：
> - **「用户发了图」≠「图生图」**。只有当用户明确要求**基于该图生成一张新图**（变风格/改背景/融合等）时才是图生图；若用户是想**了解/分析这张图**，一律交给 image 工具。判断依据是「用户想要一张**新图片**，还是想要**关于这张图的文字回答**」。
> - **「画」「图」「生成」是高频干扰字**，出现这些字时必须先确认产物是不是图像，再决定触发。

### 第三步：歧义 → 先澄清，不要闷头生成

当意图模糊到无法确定主体或方向时，**先用一句话澄清，不要直接生成一张随机图**：

| 用户输入 | 处理 |
|---------|------|
| 「来点好看的东西」 | 反问想要什么主题/风格 |
| 「我想要个 logo」「帮我设计个吉祥物」 | 可直接按通用默认生成，或先问行业/调性——二者皆可，但**不要答非所问** |
| 「给我做个表情包」 | 倾向触发，但先确认表情包上的文字内容 |
| 「（附图）这张能不能再好看点」 | 可理解为图生图优化并生成，或先问想优化哪里 |

> 歧义场景没有唯一答案，原则是：**信息足够 → 用合理默认直接生成；信息不足 → 一句话澄清。两者都比「生成一张用户没想要的图」好。**

## 调用方式

> **所有平台统一使用 `--key=value` 参数模式**，macOS / Linux / Windows 命令完全一致。

### ⚠️ 调用前必须输出提示（MANDATORY）

**在执行脚本之前**，必须先向用户输出：

```
> 🎨 **正在为你生成图片...**
> 生图通常需要 20-60 秒，请稍候。
```

### 命令格式

> ⚠️ **路径规则**: `<SCRIPT_PATH>` 已被框架替换为本 Skill 目录的**完整绝对路径**（以 `/` 开头）。直接复制使用即可，**禁止**用 `~` 或相对路径替代。

**文生图（text-to-image）**：
```bash
node "<SCRIPT_PATH>/scripts/generate.cjs" --prompt="图片描述" [--resolution=宽:高] [--revise=1|0] [--seed=数字]
```

**图生图（image-to-image）**：
```bash
node "<SCRIPT_PATH>/scripts/generate.cjs" --prompt="风格/改动描述" --images="参考图路径或URL" [--resolution=宽:高] [--revise=1|0] [--seed=数字]
```

### 参数说明

| 参数 | 必填 | 说明 | 默认值 |
|------|------|------|--------|
| `--prompt` | ✅ | 图片描述（中英文均可），最大 1024 字符 | — |
| `--images` | ❌ | 参考图片（传入即为图生图模式）。支持本地路径、URL、逗号分隔多张（最多3张） | — |
| `--resolution` | ❌ | 图片分辨率（宽:高，冒号分隔） | `1024:1024` |
| `--revise` | ❌ | 开启 prompt 智能改写优化（1=开, 0=关） | `1` |
| `--seed` | ❌ | 随机种子（>0 整数），相同 seed+prompt 可复现 | 随机 |

> **关键词含空格时**：用双引号包裹值部分，如 `--prompt="a cat in space"`
> **多张参考图**：用逗号分隔，如 `--images="/path/1.png,/path/2.png"`

## Prompt 构造规范（CRITICAL — 决定生图质量）

### 核心原则

**提示词质量直接决定生图效果。** 你必须在传入脚本前对 prompt 进行丰富和优化。

### 构造公式

一个高质量的 prompt 应包含：

```
[主体] + [场景/环境] + [风格/画风] + [光影/色调] + [构图/视角]
```

### 扩写决策表

| 用户输入特征 | 操作 |
|-------------|------|
| ≤ 5 字且无风格指定 | **必须扩写**：补充主体细节 + 场景 + 风格 + 光影 |
| 6-20 字，有主体但无风格 | **建议扩写**：补充风格和光影描述 |
| > 20 字且已含风格描述 | **直接使用**：用户意图已明确 |
| 明确说"不要改写" | 传 `--revise=0` 并使用原始 prompt |

### 扩写示例

| 用户说 | 你传入的 prompt |
|--------|----------------|
| "画只猫" | "一只橘色英短猫咪慵懒地趴在阳光充足的窗台上，柔和暖色光影，摄影风格，浅景深，4K 高清" |
| "赛博朋克" | "赛博朋克城市夜景，霓虹灯映照雨后街道，反光水面，高对比度，电影感广角构图，未来科幻风格" |
| "画个logo" | "简约扁平化 logo 设计，几何图形组合，渐变蓝紫色彩，纯白背景，矢量风格，干净利落" |
| "山水画" | "中国传统山水画，远山如黛近水含烟，松树点缀山间，留白意境，水墨风格，宣纸质感" |
| "一个女孩" | "一位年轻女性站在樱花树下，微风吹拂发丝，柔和自然光，日系清新摄影风格，浅景深虚化背景" |

### 风格关键词参考

## 图生图 (Image-to-Image) 使用指南

### 触发条件

当用户提供了参考图片（本地路径或 URL）并要求基于该图生成新图时，使用图生图模式。

### 判断逻辑

| 用户行为 | 模式 |
|---------|------|
| 纯文字描述，无参考图 | 文生图 (text_to_image) |
| 提供了图片路径/URL + 描述 | 图生图 (image_to_image) |
| 说"参考这张图"、"基于这张图"、"以图生图" | 图生图 (image_to_image) |
| 提供了图片，但要求是**理解/分析/识别图内容** | **不触发本技能** → 交 image 工具 |

### `--images` 参数格式

| 输入形式 | 示例 |
|---------|------|
| 本地文件路径 | `--images="/Users/xxx/photo.png"` |
| HTTP/HTTPS URL | `--images="https://example.com/ref.png"` |
| 多张逗号分隔 | `--images="/path/1.png,/path/2.png"` |

### 图生图 Prompt 规范

图生图的 prompt 应侧重**风格变化**和**改动方向**，而非重新描述原图内容：

| 用户说 | 你传入的 prompt |
|--------|----------------|
| "把这张照片变成油画风格" | "将参考图主体转化为古典油画风格，厚重笔触，暖色调光影，画布纹理质感" |
| "变成赛博朋克" | "参考图片主体姿态，改为赛博朋克城市背景，霓虹光影，高对比度科幻风格" |
| "变成动漫风格" | "保持参考图构图和主体，转换为日系动漫插画风格，赛璐璐着色，清晰线条" |
| "加上雪景背景" | "保持参考图主体，背景替换为冬日雪景，柔和冷色调，自然光摄影风格" |

## 分辨率选择

| 比例 | 分辨率 | 适用场景 |
|------|--------|---------|
| 1:1 | `1024:1024` | 头像、社交配图（默认） |
| 4:3 | `1024:768` | 横构图、PPT 配图 |
| 3:4 | `768:1024` | 竖构图、海报 |
| 16:9 | `1344:768` | 风景、桌面壁纸 |
| 9:16 | `768:1344` | 手机壁纸、人像 |

**自动选择策略**：
- 用户未指定 → `1024:1024`
- 提及"竖图"/"壁纸"/"人像" → `768:1344`
- 提及"横图"/"风景"/"桌面" → `1344:768`
- 提及"海报" → `768:1024`

## 输出处理

### 成功时

脚本 stdout 输出 JSON：
```json
{
  "success": true,
  "message": "✅ 图片已生成",
  "images": ["/path/to/workspace/generated-images/img_abc123.png"],
  "prompt": "原始 prompt",
  "revisedPrompt": "AI 优化后的 prompt（revise=1 时有值）",
  "resolution": "1024:1024",
  "taskType": "text_to_image"
}
```

下载失败时（附带可手动访问的临时 URL）：
```json
{
  "success": false,
  "message": "图片下载全部失败，已附上临时链接（有效期约1小时），可手动在浏览器中打开保存",
  "fallbackUrls": ["https://..."]
}
```

**AI 必须严格按以下格式输出，逐字复制模板，仅替换路径值（MANDATORY — 违反即为 BUG）**：

```
🎨 **图片已生成！**

- **保存路径**：`<images 数组中的路径>`

如果不满意，可以告诉我调整方向（如"换成油画风格"、"背景改成雪山"），我重新生成。
```

> ⚠️ **输出规则（CRITICAL — 字面级遵守，不允许任何偏离）**：
> - 你的整条 assistant 消息**只能包含**上面模板的内容，不多不少
> - **NEVER** 添加模板之外的任何额外文字、描述、解释、总结
> - **NEVER** 用自己的话描述生成了什么图片（如"已按照XXX风格生成"）
> - **NEVER** 输出 `MEDIA:` 前缀标记（框架自动处理，AI 不应输出）
> - **NEVER** 使用 `![](path)` markdown 图片语法输出路径
> - 将 `<images 数组中的路径>` 替换为脚本返回的实际绝对路径即可
> - 如果有多张图片，每张一行 `- **保存路径**：\`路径\``

## 文件命名与路径处理（CRITICAL）

- 图片文件名由 `generate.cjs` 随机生成（`img_<16位hex>.png`），无规律可循
- **stdout JSON 的 `images` 数组是唯一权威来源**，必须逐字复制，不准编造、拼接、语义化重命名
- 多步骤流程中，每一步的 `--images` 和回复路径都从对应步骤的 stdout 取，不得遗漏

---

### 失败时

脚本输出 JSON：
```json
{"success": false, "message": "错误描述"}
```

**AI 必须严格按以下格式输出失败消息（MANDATORY — 违反即为 BUG）**：

```
❌ **图片生成失败**

- **原因**：<将脚本返回的 message 如实填入>
- **建议**：<根据下方错误处理表选择对应建议>

如需重试，请再次告诉我你想画什么。
```

> ⚠️ **失败输出规则（CRITICAL）**：
> - 你的整条 assistant 消息**只能包含**上面模板的内容，不多不少
> - **NEVER** 省略模板、只回复一句话就结束
> - **NEVER** 吞掉错误信息不告知用户（即使 message 看起来是技术性错误）
> - **NEVER** 让用户卡在无回复状态——脚本返回 `success: false` 时必须立即输出上面的失败模板
> - **NEVER** 用自己的话重新解释错误原因，直接使用 message 原文

## 错误处理

| `message` 关键词 | 建议文案 |
|-----------------|---------|
| "未登录" / "未配置" | 请先登录后再使用生图功能 |
| "审核" / "不合规" / "content_rejected" | 请修改描述内容后重试，避免违规内容 |
| "超时" / "连续.*失败" | 生图服务暂时响应缓慢，请稍后重试 |
| "频繁" / "rate" | 请求过于频繁，请等待片刻再试 |
| "不可用" / "unavailable" / "网络" | 生图服务暂不可用，请稍后重试 |
| 其他 / 无法匹配 | 请稍后重试，若持续失败请联系支持 |

## 禁止行为

- **NEVER** 在输出模板之外自由发挥添加额外内容
- **NEVER** 生成涉及暴力血腥、色情露骨、政治敏感等违规内容的 prompt
- **NEVER** 生成**伪造证件/发票/公章**等可用于欺诈的图像
- **NEVER** 生成针对真实人物的 **deepfake、不雅或损害名誉**的图像
- **NEVER** 仿冒他人**肖像、品牌商标**用于误导用途
- **NEVER** 在用户未表达画图/生图意图时主动调用生图脚本
- **NEVER** 忽略脚本返回的错误信息（必须如实传达给用户）
- **NEVER** 编造图片路径或伪造生成结果——所有路径必须从 `generate.cjs` stdout JSON 的 `images` 数组逐字复制，包括多步骤流程中每一步的 `--images` 参数
- **NEVER** 对同一描述连续重试超过 2 次
- **NEVER** 传入超过 1024 字符的 prompt
- **NEVER** 在 text_to_image 模式下传 --images 参数
